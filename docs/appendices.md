# VNext Appendices

只在需要时再读这些规则，不要默认常驻加载。

## 1. 音乐 / 歌词 / OCR

- 先完成完整画面分析，再做歌词或字幕校正。
- 音乐类视频不要默认信任语音转写，优先使用更可靠字幕来源。
- OCR 必须逐场景逐张处理，不能把多张图拼在一起再识别。
- 如果拿不到可靠歌词，宁可留空或标注不可靠，也不要硬写错误文本。

## 2. 平台差异

- Bilibili / YouTube 继续走现有下载链路。
- Windows 上如果 `yt-dlp` 命令名不可见，仍然先走本地 pipeline，并回退到当前 Python 环境里的 `yt_dlp` 模块；不要因此改走浏览器、录屏或手动下载。
- 如果系统里有 Node，本地 YouTube 链路优先让下载器直接使用它；不要让宿主 agent 自己发明额外参数。
- 如果本地链路已明确报 `HTTP Error 429`，应视为平台限流阻塞，直接回报，不再擅自改路。
- 抖音公开链接优先走本地下载脚本，不把 cookie 当默认前提。
- 平台特殊问题只在遇到该平台时再加载，不放进常驻主流程。

## 3. 视听路由

- 只有进入 `finalize` 生成视听报告时，才需要加载完整路由规则。
- `finalize` 现在会顺手生成 `*_classification_summary.md/json`，优先读这份轻量摘要，不要默认回到整份长分镜表。
- 这份摘要直接来自结构化结果，不额外跑模型，默认把全片切成约 21 组前 / 中 / 后分段信号。
- `classification_result.json` 才是正式生效的路由结果；后续视听报告应优先使用它，不再把旧内置路由当作主来源。
- 先判断视觉主体和听觉主体，再决定报告框架。
- 难归类时要明确写清楚为什么难归类，不能只给空泛总结。

## 4. 恢复与异常

- 准备阶段完成后，以 `scene_scores.json` + `run_state.json` 为续跑入口。
- 批次阶段完成后，以 `host_batches/index.json` + `batch-XXX-output.json` 为续跑入口。
- `run_state.json` 只服务于“知道现在做到哪一步”，不替代完整结果文件。
- 真正交付是否成功，以 `delivery_report.json` 和实际输出文件为准。
- 如果状态文件和实际结果不一致，以已落盘的结果文件为准，并重跑一次 `score_batches` 或 `merge_validate` 进行校正。
- `merge_validate` 现在也是资源修复闸门：会优先把旧的相对路径补成完整路径，并尝试补抽缺失截图/样本图。
- OpenClaw 主控优先看 `scripts/openclaw_dispatch.py` 的输出，不要自己猜下一步派单逻辑。
- 当前批次如果 `receipt.status` 已写成 `blocked`，必须先修这一批，再继续后面的阶段。
- 如果截图、样本图或当前批次任务包缺失且无法自动修复，状态应直接进入 `blocked`，不能继续往后跑。
- 只要用户请求包含 `pdf`，任何一个必需 PDF 缺失都应算阻塞，不再只是提醒。
